# POS · Kitchen

The chef's board. It shows every open chef ticket generated by the Orders page and lets the kitchen move them through three lanes:

- **Received** — chef has not yet accepted the ticket.
- **In Progress** — chef accepted; cooking.
- **Completed** — everything ready / handed off.

Top of the page: a segmented control `Received N  ·  In Progress N  ·  Completed N` plus a sort menu (`Oldest` visible in `Cook In Progress.png`). Left drawer: scope selector (`All Floors`, per‑floor checkboxes, per‑table checkboxes) so a chef station can restrict the feed to the tables it is responsible for.

Screens used for this analysis:

- `Load Floors, Tables, select tables to show assigned tickets.png`
- `Order Received.png`
- `Cook In Progress.png`
- `Cook Completed.png`

---

## 1. Page mount

| Fetch | Method | Endpoint | Query |
|---|---|---|---|
| Floors + tables (same feed Floor Plan uses) | `GET` | `/restaurants/{rid}/floors?include=tables` | — |
| Active chef tickets | `GET` | `/restaurants/{rid}/chef-tickets` | `?status=received,in_progress,completed&tableIds=...&sort=oldest` |

Realtime updates via websocket (`chef-ticket.created`, `chef-ticket.updated`) keep the three lanes in sync without refresh. The bottom nav badge on "Kitchen" mirrors `received.count`.

## 2. Scope drawer

**Screen:** `Load Floors, Tables, select tables to show assigned tickets.png` — left rail titled *My Tables*:

- `All Floors  23/23` (master toggle).
- Per floor groups: `1st Floor 5/5`, `2nd Floor 15/15`, `Bar 3/3` — expandable, each table row has its own checkbox with a coloured dot.

### Behavior

- Selection is persisted per user (`chef` accounts typically pin their station's tables).
- Any change triggers a re‑fetch of chef tickets filtered by `tableIds`.
- The three lane counters at the top reflect the *filtered* view.

Backend contract:

```
PATCH /users/me/kitchen-scope
{ floorIds: [...], tableIds: [...] }

GET /restaurants/{rid}/chef-tickets?tableIds=T1,T2,...&status=received,in_progress,completed
```

## 3. "Received" lane

**Screen:** `Order Received.png` — four cards laid out as a row:

Each card header: `Order 10:28` (order timestamp), `• 24m elapsed` (live counter in red). Body: table name (`Table 12`, `Table 4`, `Table 10`, `Bar 2`), primary blue button **Accept**, then the list of items with their quantities (e.g. `Chow Mein 1`, `Hot & Sour Soup 2`, `Coca‑Cola 3`), and a secondary **Order Details** button.

### Behavior

- Tickets appear instantly when Orders → *Send to Kitchen* fires (`chef-ticket.created` event).
- `24m elapsed` is computed client‑side from `ticket.createdAt`.
- **Accept** moves the card to In Progress:

```
POST /chef-tickets/{id}:accept
→ { id, status: "in_progress", acceptedAt }
```

- **Order Details** opens a modal with the full order context (table, waiter, all items including those previously sent on older chef tickets, notes, allergens).

---

## 4. "In Progress" lane

**Screen:** `Cook In Progress.png` — cards have the same header shape but the primary button becomes **Complete** and each item row has a checkbox on its right side.

### Behavior

- The chef ticks items off one by one as they plate. Two visual affordances in the screenshot:
  - **Filled green check** + strike‑through name (e.g. `Pad Thai`, `White Rice Cake`, `Mango Juice`) → the item is marked ready.
  - **Empty circle** (e.g. `Green Curry`, `Seafood Pasta`, `Kung Pao Chicken`) → still cooking.
  - **Faded / dashed** rows (e.g. `Carbonara Pasta` on Table 3) → item voided by the waiter from the Orders screen after the ticket was sent. The chef should ignore it.
- Ticking an item calls:

```
PATCH /chef-tickets/{id}/items/{iid}
{ status: "ready" }
```

- **Complete** finalizes the whole ticket and must be blocked if any non‑voided item is still not `ready`:

```
POST /chef-tickets/{id}:complete
→ { id, status: "completed", completedAt }
```

The ticket moves to the Completed lane and the waiter is notified so they know the order is ready to be served.

---

## 5. "Completed" lane

**Screen:** `Cook Completed.png` — the cards show the order time and table, all items with green checks and strike‑through (entire card is "done"), and the primary button becomes **Recall**.

### Behavior

- **Recall** undoes a premature completion. It is the only way to re‑open a ticket back into In Progress:

```
POST /chef-tickets/{id}:recall
→ { id, status: "in_progress", acceptedAt: <original>, completedAt: null }
```

- Completed tickets age out of the board after a configurable period (e.g. 30 min) — they remain visible in Analytics · History forever.

---

## 6. Cross‑module notifications

- Kitchen triggers a **"Order ready"** push to the waiter when a ticket transitions to `completed`. The POS shows it as a toast (observed in `Cook Completed.png`'s top‑right: unrelated *"FLOOR PLAN · 10:53 AM — New reservation request"* toast, which demonstrates the same toast channel carrying events from any module).
- Settings · Security & Payments → Notifications has a toggle "Kitchen — New chef tickets created" that, when enabled on *other* roles (like a manager), makes them receive the `chef-ticket.created` pushes too.

## 7. State diagram — chef ticket

```
received ──accept──▶ in_progress ──complete (all items ready)──▶ completed
in_progress ──item.ready/unready──▶ in_progress (loop)
completed ──recall──▶ in_progress
                     │
                     └──(optional) auto‑archive after N min
```

Per‑item substate:

```
pending ─ready─▶ ready
ready  ─unready─▶ pending
pending ─voided from Orders─▶ voided  (rendered faded, non‑blocking for completion)
```
