## 4. QR Arrival Verification

**Screen:** `Scheduled/Step 1 - Scan QR to verify Arrival.png`.

This is a multi-step in-restaurant flow. Stepper labels:

1. `Scan`
2. `Arrived`
3. `Dining`
4. `Bill`
5. `Pay`
6. `Review`

Current step: `Scan`.

Content:

- title `Scan Restaurant QR`
- helper `Point your camera at the QR code on your table to check in`
- camera/QR frame
- status `Scanning for QR code...`
- reservation summary: `Sakura Omakase`, date/time, `2 guests`, `Table P1 · Indoor`
- development CTA `Simulate QR Scan`
- helper `In production, your camera automatically detects the QR code`

### Behavior

1. User scans a table QR generated by POS/Floor Plan.
2. QR payload identifies restaurant + table + reservation verification nonce.
3. Server validates that the reservation matches the restaurant and time window.
4. On success:
   - reservation status becomes `arrived`
   - POS table becomes `occupied`
   - POS order is opened or linked
   - app advances to `Dining`

### Backend

```json
POST /reservations/{reservationId}:check-in
{
  "qrPayload": "signed_table_qr_payload"
}
```

Response:

```json
{
  "reservationId": "res_1",
  "status": "arrived",
  "table": { "id": "t1", "name": "Table P1", "seating": "Indoor" },
  "orderId": "ord_1"
}
```

Failure cases:

- QR expired
- wrong restaurant
- wrong table
- reservation not confirmed
- outside arrival window
- already checked in

  - `Omakase Selection (8pc)` x2 - `$170.00`
- `Running Total` -> `$212.00`
- CTA `Request Bill`
- helper `Your server will finalize the bill`

### Behavior

- The customer does not add items from this screen.
- POS waiter adds items in the restaurant POS Orders page.
- Customer app subscribes to order updates and renders the live bill.
- `Request Bill` tells staff the customer is ready to settle. POS can lock editing or require final confirmation.

### Backend / realtime

Initial fetch:

```json
GET /orders/{orderId}/guest-bill
```

Realtime:

```text
order.item.added
order.item.updated
order.item.voided
order.bill.finalized
```

Request bill:

```json
POST /orders/{orderId}:request-bill
→ { "status": "bill_requested" }
```

## 11. State Machine

```text
scheduled/requested
  ├─ restaurant approves ─────▶ scheduled/confirmed
  ├─ restaurant declines ─────▶ cancelled/declined + refund
  └─ user cancels ───────────▶ cancelled

scheduled/confirmed
  ├─ scan QR / host check-in ─▶ arrived
  ├─ no-show grace elapsed ──▶ cancelled/no_show
  └─ user cancels ───────────▶ cancelled

arrived
  └─ POS order linked ───────▶ dining

dining
  ├─ request bill ───────────▶ bill_requested
  ├─ POS finalizes bill ─────▶ bill
  └─ POS cancels order ──────▶ cancelled

bill
  └─ guest pays ─────────────▶ visited
```

## 12. Cross-App Integration With POS

This Dining flow is tightly coupled to POS:

- POS Floor Plan produces reservation approval and table QR payloads.
- POS Orders updates the customer's live bill.
- POS Payment/History receives the final paid order and receipt.
- Customer app receives websocket updates so the bill changes while the customer is dining.
